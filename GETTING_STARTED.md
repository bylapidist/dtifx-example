## I'm a designer

You don't need to run any commands. This section explains how your decisions become code constraints.

### Where your tokens live

Open `tokens/foundations.json`. Every entry is a design decision:

```json
"clr": {
  "brand": {
    "$type": "color",
    "$description": "Primary brand colour, AA contrast on light backgrounds.",
    "$value": {
      "colorSpace": "srgb",
      "components": [0.055, 0.361, 0.678]
    }
  }
}
```

The `components` array is `[red, green, blue]` on a 0–1 scale. Multiply each by 255 to get familiar values: `[14, 92, 173]` = `#0E5CAD`.

| Field | What it means |
|---|---|
| `$type` | Kind of value: `color`, `dimension`, `duration`, `fontWeight`, etc. |
| `$description` | Human-readable note — this is what appears in `DESIGN_SYSTEM.md` |
| `$value` | The actual value |
| `$extensions.lapidist.governance.owner` | Who is responsible for this token |

### How your decision becomes a CSS variable

```
Your decision               Token file              CSS output
"Brand blue = #0E5CAD"  →  clr.brand in JSON  →  --catalog-tokens-clr-brand: oklch(0.4782 0.147 254.6)
                                                           │
                                                           ▼
                                           .button { background-color: var(--catalog-tokens-clr-brand); }
```

The pipeline:
1. A developer adds your decision to `tokens/foundations.json`
2. `pnpm run dtif:build` generates `ops/artifacts/build/tokens.css`
3. Component stylesheets `@import` that file and reference the variable
4. design-lint enforces that nobody writes `background-color: #0E5CAD` directly — they must use the variable

### Reading DESIGN_SYSTEM.md

`DESIGN_SYSTEM.md` is auto-generated from the token catalog. It lists every active token with its current value and deprecation status. Run `pnpm run dtif:dscp` to regenerate it after any token change.

The `generated by` header shows when it was last regenerated and which kernel snapshot it reflects.

### Dark mode and high-contrast

Theme overrides live in `tokens/themes/dark.json` and `tokens/themes/light.json`. Each override specifies which token changes and under what conditions:

```json
{
  "$token": "#/cmp/btn/bg",
  "$when": { "theme": "dark" },
  "$value": { "colorSpace": "srgb", "components": [0.839, 0.871, 0.925] }
}
```

The dark-theme button uses a light periwinkle background with dark text — maintaining contrast on a dark canvas. See [docs/theming.md](docs/theming.md) for how to add a new theme or override.

### Governance — who owns each token?

Every token carries ownership information:

```json
"$extensions": {
  "lapidist.governance": {
    "owner": "Design Foundations Guild",
    "reviewCadence": "quarterly",
    "sla": "5 business days"
  }
}
```

The audit step (`pnpm run dtif:audit`) fails if any token is missing an owner. See [docs/governance.md](docs/governance.md) for the full governance model.

## I'm a developer

### Prerequisites

Check these before starting:

```bash
node --version    # must be ≥24.0.0
pnpm --version    # must be ≥10.0.0
```

If `pnpm` is not installed: `npm install -g pnpm`
If your Node version is wrong: use `nvm use` (if you have an `.nvmrc` file) or install via [nodejs.org](https://nodejs.org).

### First run

```bash
git clone https://github.com/bylapidist/dtifx-example.git
cd dtifx-example
pnpm install          # installs all dependencies
pnpm run kernel:start # start the DSR kernel and seed tokens
pnpm run verify       # ESLint + design-lint — must exit 0
```

A clean `verify` output ends with:
```
[OK] src/components/Button.jsx
[OK] src/components/button.css
…
N problems (0 errors, N warnings)
```

Warnings about unused tokens are expected in this minimal example. **Zero errors is the goal.**

### Understanding the DSR kernel

The [DSR kernel](docs/glossary.md#dsr-kernel) is a background process that holds all tokens in memory. design-lint queries it at lint time — instead of reading token files on every run, it asks the kernel "does `rgba(14,92,173,0.4)` match any registered shadow token?"

**The kernel must be running before any `design-lint` command.** Without it, you'll see a connection refused error.

```bash
pnpm run kernel:status  # check if it's running
pnpm run kernel:stop    # stop it
pnpm run kernel:start   # restart it (required after token file changes)
```

The kernel reads `tokens/catalog.tokens.json` once at startup. **After any token change, restart the kernel.** It does not watch files.

### Making a change to an existing component

```bash
# 1. Edit the file
# 2. Run verify immediately
pnpm run verify
# 3. Fix any errors (not warnings)
# 4. Commit
```

If you see `design-token/colors` errors, you've added a raw color. Replace it with `var(--catalog-tokens-clr-*)`. See [docs/design-lint.md](docs/design-lint.md) for how each rule works.

### Making a token change

Full procedure in [docs/how-to-add-a-token.md](docs/how-to-add-a-token.md). The short version:

```bash
# 1. Edit tokens/foundations.json AND tokens/catalog.tokens.json
# 2. Restart kernel
pnpm run kernel:stop && pnpm run kernel:start
# 3. Validate
pnpm run dtif:validate
# 4. Build artifacts
pnpm run dtif:build
# 5. Verify lint
pnpm run verify
# 6. Check diff
pnpm run dtif:diff
# 7. Commit everything — source files AND ops/artifacts/
```

### Understanding CSS variable names

Tokens produce CSS variables prefixed with their source file's stem:

| Source file | Stem prefix | Example var |
|---|---|---|
| `tokens/catalog.tokens.json` | `--catalog-tokens-` | `--catalog-tokens-clr-brand` |
| `tokens/foundations.json` | `--foundations-` | `--foundations-spacing-md` |
| `tokens/components/button.json` | `--button-` | `--button-cmp-btn-bg` |

Use `--catalog-tokens-*` as your default. Use `--foundations-*` when you want to be explicit about referencing a primitive. See [docs/architecture.md](docs/architecture.md) for why this naming exists.

### Inline lint suppression

When a violation is intentional, suppress it with a comment explaining why:

```css
/* design-lint-disable-line design-token/animation, design-token/duration */
animation: spin 1s linear infinite;
```

```jsx
// design-lint-disable-next-line design-system/component-usage
// Card.jsx is the DS implementation of <article>
<article className="card">
```

```jsx
{/* design-lint-disable design-system/component-prefix */}
{/* Documented exception: legacy entry point, tracked in DS-412 */}
<Button>Legacy</Button>
{/* design-lint-enable design-system/component-prefix */}
```

**Suppression is appropriate** for: implementing a DS component (suppressing its own rule), documented legacy exceptions, third-party values not in the token catalog.
**Suppression is not appropriate** for: working around a violation you don't understand, making CI pass without fixing the issue. See [docs/design-lint.md](docs/design-lint.md#inline-lint-suppression) for the full guide.

## I'm setting up CI / platform engineering

### What the pipeline does

See [docs/ci-pipeline.md](docs/ci-pipeline.md) for the full step-by-step breakdown. In summary:

| Phase | Steps | What they check |
|---|---|---|
| Setup | install, kernel start | Dependencies and token loading |
| Lint | validate, verify, warning budget, SARIF | Code quality against token rules |
| Artifacts | JSON report, DSCP export, snapshot diff | Documentation and change detection |
| Token pipeline | dtif:validate, build, diff, audit | Schema conformance, artifact consistency, governance |
| Cleanup | kernel stop | Daemon teardown |

### The diff baseline

`ops/artifacts/diff/baseline.dtif.json` is the committed snapshot of approved tokens. `dtif:diff` compares the current catalog against it and highlights breaking changes (removed tokens, type changes).

**To update the baseline after an intentional token change:**
```bash
cp tokens/catalog.tokens.json ops/artifacts/diff/baseline.dtif.json
git add ops/artifacts/diff/baseline.dtif.json
git commit -m "chore: update diff baseline after adding X token"
```

### The kernel snapshot baseline

`ops/artifacts/snapshots/baseline.bin` is a binary snapshot of the kernel's token graph. `design-lint:snapshot-diff` exports the current graph and diffs against this file to detect token changes between runs.

**To update the snapshot baseline:**
```bash
pnpm run kernel:start
pnpm run design-lint:snapshot
cp ops/artifacts/build/kernel-snapshot.bin ops/artifacts/snapshots/baseline.bin
git add ops/artifacts/snapshots/baseline.bin
git commit -m "chore: update snapshot baseline"
```

### The warning budget

`design-lint:check` runs with `--fail-on-empty --max-warnings 100`. The warning budget prevents silently-empty globs from passing and caps accumulated warnings at 100. The current warning count (mostly `no-unused-tokens` for tokens not referenced in the minimal example) is well within budget.

### The ratchet policy

`designlint.policy.json` enforces a minimum entropy score of 80/100. If a PR worsens the design system health score (by adding violations), CI fails even if no existing errors were introduced. This prevents gradual decay.

**The entropy score** reflects how consistently the codebase uses design tokens. More violations → lower score. To improve it: fix lint errors and reduce warnings.

### Adding design-lint to an existing project

```bash
# 1. Install
pnpm add --save-dev @lapidist/design-lint

# 2. Generate a starter config
pnpm exec design-lint init

# 3. Start the kernel and seed with your token catalog
pnpm exec design-lint kernel start --config-path designlint.config.json

# 4. Run once with warnings-only to establish a baseline
pnpm exec design-lint "src/**/*" --max-warnings 9999

# 5. Promote rules to error as your team adopts the system
```

Start with the `recommended` preset and tighten over time. Copy the relevant CI steps from `.github/workflows/ci.yml` into your pipeline. See [docs/ci-pipeline.md](docs/ci-pipeline.md) for what each step does and how to handle failures.
