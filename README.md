# DTIFx Minimal Stack Example

A working example of a production design-token pipeline. It shows how design decisions made in Figma (or a spreadsheet) become enforced code constraints — so developers cannot accidentally use a colour, spacing value, or animation timing that the design system hasn't approved.

If you're new here, start with **[GETTING_STARTED.md](GETTING_STARTED.md)**.

---

## What problem does this solve?

| Without this stack | With this stack |
|---|---|
| Developers hard-code `#0E5CAD` in CSS | Lint fails: "use `var(--catalog-tokens-clr-brand)`" |
| Token values drift between design and code | Single source of truth: `tokens/` folder |
| No one notices a deprecated colour is still used | CI fails: `design-system/deprecation` rule |
| Governance is informal | `ops/artifacts/audit/` contains dated evidence |

---

## Glossary

Before reading further, these terms appear everywhere:

| Term | Plain English |
|---|---|
| **Design token** | A named value (`clr.brand = #0E5CAD`) that both designers and developers agree to use |
| **DTIF** | Design Token Interchange Format — the JSON schema tokens are written in |
| **DTIFx** | The CLI toolchain that validates, builds, diffs, and audits DTIF tokens |
| **design-lint** | The linter that checks your CSS and JSX files against the token catalog |
| **DSR kernel** | A background process that holds all tokens in memory so the linter can query them fast |
| **DSQL** | The internal query language the kernel uses — you never write it directly |
| **Catalog** | `tokens/catalog.tokens.json` — the single file that aggregates every token source |
| **Artifact** | A file generated by a build command and committed to `ops/artifacts/` for review |
| **Governance** | Rules about who owns a token, how often it's reviewed, and what the SLA is |

## Architecture

```
tokens/foundations.json          ← primitive values (colours, spacing, …)
tokens/components/button.json    ← component-specific tokens
tokens/themes/{light,dark}.json  ← contextual overrides
        │
        ▼ pnpm run dtif:build
ops/artifacts/build/
  tokens.css   ← CSS custom properties  (:root { --clr-brand: oklch(…); })
  tokens.json  ← JSON snapshot for tooling
        │
        ▼ @import in component stylesheets
src/components/button.css        ← uses var(--catalog-tokens-clr-brand)
src/components/Button.jsx        ← renders the button
        │
        ▼ pnpm run design-lint
design-lint checks every var() reference exists in the running DSR kernel.
Hardcoded colors, spacing, or animations → lint error → CI fails.
```

## Quick start

```bash
# 1. Install
pnpm install

# 2. Start the token kernel and verify everything passes
pnpm run kernel:start
pnpm run verify

# 3. Build token artifacts and check for drift
pnpm run dtif:build
pnpm run dtif:diff
```

That's it for day-to-day use. See [GETTING_STARTED.md](GETTING_STARTED.md) for audience-specific paths.

## Repository layout

| Path | Purpose |
|---|---|
| `tokens/` | Token source files — the single source of truth |
| `src/` | Example UI components that consume the tokens |
| `ops/artifacts/` | Generated outputs committed for review (CSS, JSON, SARIF, audit reports) |
| `ops/scripts/` | Node scripts for programmatic API demos |
| `ops/plugins/` | Custom design-lint rule plugin |
| `ops/formatters/` | Custom design-lint output formatter |
| `ops/policies/` | Governance policy files |
| `ops/snapshots/` | DSR kernel snapshots for drift detection |
| `build/` | DTIFx build and validate configuration |
| `audit/` | DTIFx audit policy configuration |
| `docs/` | Guides — token editing, script reference, architecture |
| `designlint.config.js` | Which lint rules are active and at what severity |
| `designlint.policy.json` | CI governance constraints (required rules, ratchet mode) |

## Scripts — grouped by audience

Full reference: [docs/script-reference.md](docs/script-reference.md)

### Every day
```bash
pnpm run kernel:start     # start the token kernel (required before linting)
pnpm run verify           # ESLint + design-lint (must pass before every commit)
pnpm run dtif:build       # regenerate tokens.css and tokens.json
```

### When editing tokens
```bash
pnpm run dtif:validate    # check token files against the DTIF schema
pnpm run dtif:diff        # compare against the committed baseline
pnpm run dtif:audit       # run governance policies
```

### CI / reporting
```bash
pnpm run design-lint:sarif     # SARIF report for GitHub code scanning
pnpm run design-lint:validate  # validate config and active rules
pnpm run dtif:dscp             # regenerate DESIGN_SYSTEM.md
```

## Troubleshooting

**"DSR kernel failed to start" or connection refused**
The kernel must be running before any lint command. Run `pnpm run kernel:start` and check `pnpm run kernel:status`.

**"No tokens configured" — rules pass when they should fail**
The kernel started without tokens. Stop and restart: `pnpm run kernel:stop && pnpm run kernel:start`.

**`dtif:diff` reports unexpected changes**
Your local token files have diverged from the committed baseline. Run `pnpm run dtif:build` then review `ops/artifacts/diff/report.json`.

**`design-lint:validate` fails with "Unknown rule"**
A rule in `designlint.config.js` isn't registered. If it's from the custom plugin (`dtifx/…`), remove it from `rules:` (the plugin is loaded via `plugins:` and doesn't need an explicit rules entry for validate to pass).

**Lint passes locally but fails in CI**
CI starts the kernel explicitly. If you skipped `kernel:start` locally, re-run `pnpm run kernel:start` then `pnpm run verify`.
