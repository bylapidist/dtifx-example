# Glossary

Every term used across this repository, in one place. Terms link back to the document where the concept is explained in depth.

---

**Artifact**
A file generated by a CLI command (`dtif:build`, `dtif:audit`, `design-lint:sarif`, etc.) and committed to `ops/artifacts/` so PR reviewers can inspect changes without re-running tools. See [ops/README.md](../ops/README.md).

**Catalog** (`catalog.tokens.json`)
The single aggregator file that inlines every token from every source (`foundations.json`, `components/`, `themes/`). The [DSR kernel](#dsr-kernel) reads this file at startup. See [docs/architecture.md](architecture.md).

**CSS custom property**
A CSS variable declared with `--` prefix (e.g. `--foundations-spacing-md: 16px`). Generated from token files by `dtif:build` and consumed in component stylesheets via `var(--...)`. See [docs/token-system.md](token-system.md).

**DSCP** (Design System Context Package)
The format produced by `dtifx dscp generate` and stored in `DESIGN_SYSTEM.md`. A structured Markdown document describing every token, rule, and component — consumed by MCP tools and AI assistants.

**DSR kernel** (Design System Runtime kernel)
A long-lived Node.js daemon (`design-lint kernel`) that holds the authoritative token graph in memory. Every `design-lint` invocation queries the kernel via a Unix socket rather than re-reading token files. See [docs/design-lint.md](design-lint.md).

**DSQL** (Design System Query Language)
The internal protocol the DSR kernel uses to answer queries from the linter. You never write DSQL directly; design-lint uses it internally to check whether a CSS value matches a token.

**DTIF** (Design Token Interchange Format)
The JSON schema all token files in this repo follow. Based on the W3C Design Tokens Community Group draft, extended with richer type information and JSON Pointer references. See [docs/token-system.md](token-system.md).

**DTIFx**
The CLI toolchain (`dtifx`) that validates, builds, diffs, and audits DTIF token files. Distinct from design-lint, which lints source code. See [docs/architecture.md](architecture.md).

**Design token**
A named design decision stored as a value: `clr.brand = #0E5CAD`. Tokens let designers and developers share a single source of truth. See [docs/token-system.md](token-system.md).

**Entropy score**
A 0–100 measure of design system health used by the ratchet policy. Affected by the number and severity of lint violations. The policy requires the score to stay above a minimum threshold so CI fails if a PR worsens it. See [docs/ci-pipeline.md](ci-pipeline.md).

**Governance**
Metadata attached to each token (`$extensions.lapidist.governance`) declaring who owns it, how often it's reviewed, and the SLA for changes. Enforced by `dtif:audit`. See [docs/governance.md](governance.md).

**JSON Pointer**
A string that uniquely identifies a token within a DTIF document, e.g. `#/clr/brand`. Used by design-lint rules and the DSR kernel as stable identifiers. Format: `#/` followed by the path of JSON keys separated by `/`.

**Lapidist**
The organisation that publishes DTIFx, design-lint, and related packages under `@lapidist/`. See [dtifx.lapidist.net](https://dtifx.lapidist.net/).

**Prepack**
A `package.json` lifecycle script that runs automatically before `pnpm publish`. Used by the design-lint config preset packages (`@lapidist/design-lint-config-*`) to compile TypeScript source to `dist/` before publishing.

**Provenance** (as in `design-token/css-var-provenance`)
The rule that checks every `var(--...)` reference in source code is backed by a registered DTIF token. "Provenance" = knowing where a variable came from. This rule is set to `warn` (not `error`) in this repo because our build adds a source-file prefix to var names (e.g. `--catalog-tokens-clr-brand`) that doesn't match the pointer-derived name the rule expects (`--clr-brand`). See [docs/architecture.md](architecture.md).

**Ratchet mode**
A CI policy that prevents design system health from decreasing between commits. Once a PR brings the entropy score below the minimum (`minScore: 80`), CI fails even if the total violation count is unchanged. See [docs/ci-pipeline.md](ci-pipeline.md).

**SARIF** (Static Analysis Results Interchange Format)
A JSON schema for lint results, version 2.1.0. Used by GitHub code scanning to annotate pull requests with lint violations. Produced by `design-lint:sarif`.

**Token coverage**
The percentage of registered tokens that are actually referenced in linted source files. `designlint.policy.json` requires at least 80% of color tokens and 60% of dimension tokens to be referenced.

**Token pointer** → see [JSON Pointer](#json-pointer).
